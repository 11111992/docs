What is it?

Cloud Native Buildpacks transform your application source code into images that can run on any cloud.
A Dockerfile is a text file that contains commands that will be executed by docker to build a container image.
Source-to-Image (s2i) is a toolkit and workflow for building reproducible container images from source code.
ko is a simple, fast container image builder for Go applications. It's ideal for use cases where your image contains a single Go application without any/many dependencies on the OS base image (e.g., no cgo, no OS package dependencies).
Jib builds optimized Docker and OCI images for Java applications without a Docker daemon. It is available as plugins for Maven and Gradle and as a Java library.

Reproducibility

Cloud Native Buildpacks project aims to create “Reproducible Builds” of container images. For image creation commands (builder create, buildpack package, build) pack creates container images in a reproducible fashion.
No, reproducible builds with dockerfiles come with caveats.
Yes, Source-to-image allows build environments to be tightly versioned by encapsulating them within a container image and defining a simple interface (injected source code) for callers.
Yes, Ko is a container image builder for Go applications and supports reproducible builds for Go.
Yes, Jib supports reproducible builds. Rebuilding the container image with the same contents always generates the same image. Unnecessary updates need not be triggered.

Advanced Caching

Buildpacks implement advanced caching of images, cache images are a way to preserve build optimizing layers across different host machines. These images can improve performance when using pack in ephemeral environments such as CI/CD pipelines.
No, Requires configuration in the Dockerfile, the cache relies on both the instruction being executed and the image generated from the previous instruction, changing any instruction in the Dockerfile will invalidate the cache for all of the instructions that follow it. Invalidating an image also invalidates all the children of that image.
Yes, can be configured to capture the artifacts from a previous build into the next incremental build.
No, not a feature provided by Ko.
No, not a feature provided by Jib.

Multi-Language

Auto-Detection

Cloud Native Buildpacks support auto-detection. This happens when a platform sequentially tests groups of buildpacks against your app’s source code. Detection criteria is specific to each buildpack – for instance, an NPM buildpack might look for a package.json, and a Go buildpack might look for Go source files.
No, Dockerfiles need to be written and configured according to the needs of build.
Yes, Source-to-Image supports auto detection.
Yes, by default Ko supports only the the Go programming language, ko builds images by effectively executing go build on the local machine, and as such doesn't require docker to be installed.
Yes, by default Jib supports only the the Java programming language.
